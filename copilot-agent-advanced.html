Part 1 · 认识 Agent

01

Agent 是什么 · 在 Microsoft 365 中的位置 · 与 Copilot Chat 的区别

Agent 体系总览

Source: Agent 应用地图 · Get Started with Agents · Agents in Microsoft 365

Agent 是什么？

Agent 是 Microsoft 365 Copilot 的扩展能力——它不是独立的应用，而是运行在 Copilot 生态中的智能助手。Agent 可以理解特定的业务场景、访问指定的数据源、执行预定义的任务，并在 Copilot 对话中被调用。

Agent vs Copilot Chat

维度Copilot ChatAgent
范围通用问答特定场景/任务
数据Web + 工作文件可指定数据源
创建无需创建由用户/IT创建
适用人人可用按需配置

Agent 的三种形态

  • 预构建 Agent：Microsoft 官方提供的开箱即用 Agent（如 SharePoint Agent）
  • Agent Builder Agent：用户在 Copilot Chat 中用自然语言创建的 Agent
  • Copilot Studio Agent：在 Copilot Studio 中构建的高级 Agent，支持复杂逻辑和外部连接

Word / Excel / PowerPoint Agents — 从应用内 Copilot 到应用 Agent

Source: Word Excel and PowerPoint Agents · Microsoft Support + Microsoft Learn Training

Word Agent

基于文档内容生成报告、提案、摘要。不只是"Copilot 写一段"，而是"Agent 理解整份文档并执行多步骤任务"。

Excel Agent

自动分析数据、生成洞察、创建可视化。Agent 可以记住你的分析偏好并自动应用。

PowerPoint Agent

基于文档/数据自动生成演示文稿。Agent 理解品牌风格并保持一致。

与基础 Copilot 的区别

基础 Copilot 是"单次问答"；App Agent 可以持续运行、记住上下文、跨文档工作。

适用对象：普通员工中的进阶用户、Champion/种子用户、培训讲师。不建议放入第一轮全员基础课——先让用户熟悉 Copilot 基础后再引入 Agent 概念。

SharePoint · Employee Self-Service · Analyst/Researcher · App Agents

SharePoint Agents — 基于组织知识的 Agent

Source: Agents in SharePoint · Microsoft Adoption

应用场景

SharePoint 是企业知识和文件协作的中心。SharePoint Agent 可以：

  • 回答基于 SharePoint 站点内容的问题（"公司的差旅政策是什么？"）
  • 自动归纳文件夹中的文档变化
  • 帮助新员工快速找到相关政策和模板
  • 作为项目知识库的智能检索入口

使用场景举例

  • HR 站点 → 员工自助问答 Agent
  • 项目站点 → 项目文档智能助手
  • 销售站点 → 产品知识/竞品信息检索
  • 合规站点 → 政策法规智能检索

莱茵落地建议

  • 先盘点莱茵内部已有的 SharePoint 站点
  • 选 1-2 个文档量大、查询频率高的站点做试点
  • 与 SharePoint 权限治理同步规划
  • 种子用户先测试再推广

Employee Self-Service Agent — 员工自助服务

Source: Employee Self-Service Agent in Microsoft 365 Copilot · Microsoft Adoption

应用场景

代表一种典型的企业内部服务型 Agent场景——员工不再需要翻找内网、打电话给HR/IT，而是直接向 Agent 提问。

典型问题域

  • HR：请假流程、福利查询、入职指南、培训报名
  • IT：设备申请、密码重置、软件安装、网络问题
  • 行政：会议室预订、访客登记、办公用品申请
  • 财务：报销流程、差旅标准、发票要求

莱茵落地建议

  • 适合 HR/IT/行政联合推进
  • 先做 FAQ 类问题（答案明确、数据好整理）
  • 逐步扩展到流程类问题（需要对接后端系统）
  • 可作为莱茵内部数字化转型的示范项目

Analyst / Researcher Agent + App Agents

Source: AnalystResearcherOverviewDeck.pptx + Microsoft365CopilotAppAgents Guide & Handout

Analyst / Researcher Agent — 知识工作增强型

代表"知识工作者增强型 Agent"方向。适合研究资料整理、分析报告初稿、信息归纳和业务洞察生成。对莱茵咨询、销售、项目管理、市场研究类场景有直接启发。

  • 研究资料收集与归纳 → 生成分析报告初稿
  • 竞对信息监控 → 市场动态摘要
  • 项目文档分析 → 风险识别和要点提取

App Agents — 从应用内 Copilot 到应用 Agent

代表"从 Office 应用内的 Copilot 能力走向独立 Agent"的进阶路径。Office App Agents 可以跨 Word/Excel/PPT 工作，执行多步骤任务。

  • 自动监控 Excel 数据变化 → 触发报告更新
  • 基于 Word 模板 → 自动生成定制化文档
  • 分析多个 PPT → 提取跨演示文稿的主题一致性

Agent Builder vs Copilot Studio — 选型决策

Source: Choose between Agent Builder and Copilot Studio · Microsoft Learn

核心判断标准

不是"哪个更好"，而是"你的场景适合哪个"。

维度Agent BuilderCopilot Studio
门槛低——自然语言创建中高——需要配置技能
适合谁业务用户、种子用户IT、开发者、高级制作者
数据源SharePoint、Web、文件以上 + API、数据库、外部系统
逻辑复杂度简单流程、FAQ多步骤、条件分支、外部调用
治理基本共享完整生命周期管理
典型场景"帮我回答关于 XX 项目的问题""对接ERP系统处理请假申请"

决策建议：先让种子用户用 Agent Builder 快速验证场景（1-2天出原型）。验证通过且需要更复杂能力时，再迁移到 Copilot Studio。不要一上来就推 Copilot Studio——门槛太高会吓退业务用户。

Build · Share · Manage — Agent 生命周期

Source: Build Agents with Microsoft 365 Copilot · Share and Manage Agents · Microsoft Adoption/Learn

Build — 构建 Agent

  1. 明确场景：这个 Agent 要解决什么问题？
  2. 选择工具：Agent Builder 还是 Copilot Studio？
  3. 定义知识源：SharePoint 站点/文件/Web
  4. 设定行为：Agent 应该怎么回答、不能做什么
  5. 测试迭代：用真实问题测试，调整 Prompt 和知识源

Share — 分享 Agent

  1. 从"只有我能用"到"团队/组织能用"
  2. 在 Copilot Chat 中通过链接分享
  3. 在 Teams 频道中发布 Agent
  4. 管理员可在 Admin Center 管理可见范围

Manage — 管理 Agent

  1. 谁创建了哪些 Agent？（Agent Registry）
  2. Agent 使用了哪些数据源？权限是否合理？
  3. Agent 的使用频率和效果如何？
  4. 何时需要更新、下线或替换 Agent？

常见陷阱

  • 知识源太广 → Agent 回答不精准
  • 权限未控制 → 敏感信息泄露
  • 只建不管 → 过期 Agent 给出过时答案
  • 没有反馈机制 → Agent 质量无法提升

Part 5 · Agent 治理与安全

05

Governance · Security · Registry · Evaluation

Agent Governance — 治理框架

Source: Agent-governance-whitepaper.pdf · Microsoft Adoption

核心理念

Agent 不是一次性创建物——它是需要持续治理的组织资产。就像管理应用程序一样管理 Agent。

治理覆盖维度

  • 创建：谁可以创建 Agent？需要什么审批？
  • 权限：Agent 使用哪些数据？谁授权？
  • 监控：Agent 使用频率？回答质量？
  • 评估：Agent 是否仍然有效？需要更新？
  • 退役：何时下线？如何迁移用户？

关键角色

  • 业务负责人：定义场景、审批创建、评估价值
  • IT 管理员：技术实现、权限控制、Registry 管理
  • 安全/合规：数据边界、风险评估、审计
  • 培训团队：用户教育、Champion 培养
TÜV Rheinland · Agent 治理5.1

Agent Security + Registry + Evaluation

Source: Agent-security-playbook.pdf · Agent Registry · AgentEvaluationEbook.pdf

Agent Security — 安全手册

  • 数据边界：Agent 不能访问用户无权访问的数据
  • 权限继承：Agent 的权限基于创建者和使用者的双重校验
  • 人工复核：关键决策必须有人的最终确认
  • 审计追踪：Agent 的所有操作应可追溯

Agent Registry — 资产清单

  • Microsoft 365 Admin Center 中的 Agent 管理面板
  • 让 Agent 从"个人创建物"变成"可被组织看见的资产"
  • 管理员可以查看、管理、限制 Agent 的使用范围
  • 支持按部门/角色配置 Agent 可见性

Agent Evaluation — 评估框架

  • 质量：回答准确率、相关性、完整性
  • 安全：不泄露敏感信息、不执行危险操作
  • 体验：用户满意度、响应速度、易用性
  • 价值：节省的时间、解决的工单数

莱茵落地：上线前检查清单

  • Agent 定义清晰：要解决什么问题？
  • 数据源已确认：权限范围是否合理？
  • 安全评估通过：不会泄露敏感信息？
  • 已注册到 Registry：管理员可见
  • 有反馈机制：用户如何报告问题？

Agentic Readiness · 90天社区框架 · 组织采用路径

Agentic Readiness — 组织准备度评估

Source: AgenticReadinessFrameworkOverview.pdf · Microsoft Adoption

核心问题

判断组织是否准备好从 Copilot 使用走向 Agent 采用。

评估维度

维度判断标准
数据治理组织数据是否已分类、权限是否清晰？
业务场景是否有明确的、可量化的 Agent 应用场景？
人员能力团队是否具备基本的 Copilot/AI 使用经验？
治理机制是否有 Agent 审批、监控、评估的流程？
价值衡量能否定义和追踪 Agent 的业务价值？

莱茵自评建议

  • 先完成 30 天 Copilot Journey → 建立用户基础
  • 从 Journey 完成者中识别 Agent 种子用户
  • 选 2-3 个高价值场景做试点（不要贪多）
  • 边试点边建立治理规则
  • 6 个月后做正式的 Readiness 评估

90-Day Community Framework — 种子用户社群运营

Source: 90DayCommunityFramework.pdf · Microsoft Adoption

核心理念

Agent 采用不应只依赖一次培训——需要持续社区、反馈和实践机制。90 天是一个完整的"建立→激活→自运转"周期。

第 1 个月 · 建立

  • 招募首批 Champion（从 Journey 完成者中选）
  • 创建 Teams 频道
  • 每周 1 次 Office Hour
  • 发布第 1 批 Agent 模板

第 2 个月 · 激活

  • 搞 Agent 创作大赛
  • 收集用户故事和案例
  • 跨部门分享会
  • 识别"超级 Champion"

第 3 个月 · 自运转

  • Champion 带新人
  • 建立案例库和 FAQ
  • 月度 Agent 质量评审
  • 向管理层汇报成果
