前言
这门课是怎么来的

2026年，June，从0开始搭建了自己的AI Agent——八戒。从懵懵懂懂一无所知，到后来八戒能帮她做市场情报分析、项目方案起草、竞对监测、数据分析报告……只用了不到一个月。

这个过程，被完整记录下来，就是这门课程。

课程不是教你怎么用Hermes，而是教你——怎么像June一样，从零开始为自己的岗位量身定制一个AI Agent。

🔑 配置API Key

让Agent能用大模型 · 3分钟

Agent装好了，但它还"没有大脑"——它需要连上一个能思考的大模型。API Key就是它跟大模型对话的凭证。

选择一个Provider

推荐新手用 OpenRouter——一个整合了几乎所有大模型的平台，一个Key就能用GPT/Claude/DeepSeek等几十个模型。

配置

# 新建 ~/.hermes/.env 文件
OPENROUTER_API_KEY=sk-or-v1-xxxxxxxxxxxxxxxx

选择默认模型

model: openrouter/anthropic/claude-sonnet-4
June的经验：我一开始用DeepSeek（便宜），后来换到Claude（更听话）。新手先用便宜的模型试。

实战升级：2026年6月1日 — 配置Kimi K2.6作为备用模型

运行一段时间后，June需要的不仅是单个模型的能力，还有按需切换的灵活性——DeepSeek是日常主力，Kimi K2.6作为备用模型，两者各有所长。关键认知：模型只是Agent的\"底座发动机\"，八戒（Agent层）的角色设定、技能库、记忆体系、工具调用全部独立于底座模型——换模型不会影响Agent的行为模式。

① 多模型架构

层角色说明
用户层你（June）提需求不管底层用哪个模型
Agent层八戒（角色设定+技能+记忆+工具）不绑定模型，换模型不影响
底座模型层DeepSeek V3（主力） ↔ Kimi K2.6（备用）只在\"思考+写回复\"这层切换

② 配置Kimi

在 ~/.hermes/config.yaml 和 ~/.hermes/.env 中并排配置两个provider：

# config.yaml：两个provider并排
model: deepseek-chat <- 当前活动模型
providers:
  deepseek:
    base_url: https://api.deepseek.com/v1
    api_key: ${DEEPSEEK_API_KEY}
  kimi:
    base_url: https://api.moonshot.cn/v1
    api_key: ${KIMI_API_KEY}

# .env：两个Key各放各的
DEEPSEEK_API_KEY=sk-...
KIMI_API_KEY=sk-...

③ 切换模型

切换模型只需改一行配置，然后重启Gateway生效：

# config.yaml 第3行
model: deepseek-chat <- 主力
model: kimi-k2.6 <- 切到Kimi
切换方式操作影响
手动改配置改config.yaml第3行 model: 值重启gateway后生效
切换脚本bash ~/.hermes/switch_model.sh kimi一行命令自动改配置+重启
终端（CLI）关掉旧hermes进程 → 重新 hermes几乎无感

切换脚本放在 ~/.hermes/switch_model.sh，调用sed修改config.yaml的model行，简化了手动编辑步骤。

④ 切换模型不影响什么

Agent资产存在哪里切模型会丢吗
Skill（会写竞对报告/会PPT/会分析）硬盘 D:\Hermes\skills\❌ 不会丢
Memory（记得June偏好/习惯/人名） D:\June\_wiki\ + 记忆系统❌ 不会丢
SOUL（重庆话/敢挑战/框架思维）系统提示词（人设）❌ 不会丢
TASKS（待办事项）TASKS.md 文件❌ 不会丢
会话历史SQLite数据库❌ 不会丢

真正会受影响的只有回复的风格——DeepSeek简洁直接，Kimi说话更详细。但Agent的行为模式（会搜索、会写报告、会用重庆话回你）不受任何影响。

⑤ 关键原则

🐷 八戒实战教训：
① Agent层跟模型层解耦。 换模型不会改变Agent会什么、记得什么、怎么说话。底座模型只是\"思考引擎\"，Agent的能力在自身这层。
② 没有完美的模型，多一个选项就是多一个Plan B。 DeepSeek性价比高适合日常，Kimi K2.6可以兜底。具体用哪个取决于任务类型和个人偏好。
③ 切换后需要重启Gateway（微信端）或重启进程（终端）。 配置改了不代表当前会话自动切换——新的配置在新会话中生效。

🕸️ 配置搜索能力（Tavily）

让Agent能搜实时信息 · 3分钟

Hermes装好了、能聊天了，但你一问它"最近有什么政策变化"，它就傻了——因为它没有联网能力。Tavily就是给Agent装上一个搜索引擎。

注册Tavily

打开 tavily.com，用邮箱注册。免费额度够个人日常使用。

拿到API Key

注册后在Dashboard复制你的API Key（以 tvly- 开头）。

配置到Hermes

# 打开 ~/.hermes/.env，加上这一行
TAVILY_API_KEY=tvly-xxxxxxxxxxxxxxxx

SOUL.md的自动加载——写完放哪才能生效

常见问题 · 3分钟

SOUL.md写完了不等于Agent就自动变成那个样子。你还得把文件放到正确的位置。

问题说明解决方案
文件放哪？ Hermes在启动时自动读 HERMES_HOME/SOUL.md。你的HERMES_HOME是啥路径，文件就放哪。可以通过终端运行 echo $HERMES_HOME 查看。 把SOUL.md放到 HERMES_HOME 目录下。如果这个目录里有多个Hermes配置文件（config.yaml、.env等），那就对了。
文件有隐形字符被拦截 文件开头可能有不可见字符（比如U+FEFF，即BOM标记），会被系统的安全检查误判为prompt injection，导致整个SOUL.md被阻断——Agent收不到人格设定。 用 hexdump -C SOUL.md | head -3 检查文件头。如果第一行出现 ef bb bf（UTF-8 BOM），用编辑器（VS Code/Vim）另存为「UTF-8 无BOM」格式即可修复。
修改后没生效 系统提示词在会话启动时一次性注入。你正在聊的对话里改了SOUL.md，Agent的人格不会立刻变——得关掉重开一个新会话。 修改SOUL.md后，发 /new 指令（CLI）或关掉微信对话重开。新会话启动时会重新读SOUL.md，注入最新人格。
多个SOUL.md文件混淆 有些系统会自动在 ~/.hermes/ 下生成SOUL.md作为默认模板，但实际生效的是 HERMES_HOME 下的那个。两处文件内容不一致时，会搞不清Agent为啥没有按预期工作。 只维护 HERMES_HOME 下的SOUL.md，把其他位置的同名文件删掉或重命名。统一一个编辑点，避免混乱。
八戒实战教训：一开始SOUL.md有两个版本——C盘 ~/.hermes/SOUL.md（67行）和D盘 HERMES_HOME/SOUL.md（71行）。D盘才是系统实际读的那个，C盘是历史遗留的模板。更糟的是文件开头有个U+FEFF隐形字符，导致系统启动时安全检查拦截了SOUL.md，自动加载失败但没有任何报错提示。折腾了两天才找到原因。

正确做法：写完SOUL.md后，第一时间去验证——关掉当前会话开新会话，看Agent说话风格、行为规则是不是你设定的样子。是的话说明加载成功；不是的话，检查文件位置和文件头有无隐形字符。

Step 3 · 流程文件

怎么把做法固化成可执行的SOP？

天齐项目在这一步输出了SOP、作业指导书(WI)、表单模板、KPI体系、岗位说明书。对应到June的工作，就是把乌龟图的分析结果写成可执行的流程文件。

SOP（标准作业程序）
每个子过程的执行步骤、谁在什么时候做什么、输入输出标准。比如S1的SOP：每周一上午八戒采集六大方向情报→June周二前确认关键信号→周三输出简报。
KPI体系
每个过程的衡量指标。结果指标（合同按时率、预算达成率）+ 引领指标（采购申请处理时间）。对应到June：机会转化率、方案提交及时率、客户满意度。
岗位说明书
基于KPI导向，描述核心工作内容、所需能力、产出标准。June自己的岗位说明书就是USER.md的升级版——把18个子过程的能力要求写进去。
表单模板
每周用的周报模板、客户画像模板、方案框架模板、报价模板。这些模板让每个过程的执行标准化——June不需要每次都从头想格式。
流程文件的意义：没有流程文件，每个过程都靠人脑记忆。有了SOP，新人能快速上手；有了KPI，效果能被衡量；有了模板，效率能提升。流程文件也是后面让Agent介入的基础——有标准流程的事，才适合交给Agent。

关键实战教训（今天踩的坑）

做数据分析报告最容易翻车的地方
🔴 错误：用MarkItDown读Excel
MarkItDown只能把Excel转成文本表格，不能做groupby汇总。必须用Python pandas做数据聚合后再分析。
🔴 错误：在同一个response里做太多操作
推理文字 + write_file(30KB+) + read_file(大文件) 三者合起来超出输出token上限 → 报\"Response truncated\"。解决办法：用execute_code一次构建完整HTML写入。
🔴 错误：patch分段写HTML
先write_file前半再用patch追加后半，容易破坏script标签和文件尾部的</body></html>。应该用execute_code构建完整HTML + write_file一次性写入。
🔴 错误：f-string拼接Chart.js
Python f-string里用{}拼接Chart.js脚本——花括号被解析为f-string占位符。用字符串+拼接或在外部构建好再write_file。
June的经验：数据报告最容易错的不只是图表渲染，还有数字本身。每次更新数据后，必须做交叉验证——用Python脚本对比新提取的数据和旧报告中的数字，差异¥500以内的舍入差可以接受，超过就要追查原因。

7.4 HTML→PDF踩坑全集（15版血泪史）

一次真实的问卷报告HTML→PDF迭代，从v1到v15，所有踩过的坑和系统性解决方案

背景：2026年5月27日，八戒为BS.A West团队生成一份调研问卷分析报告的PDF。同一个HTML文件，从v1改到v15，才最终输出一份格式正确的PDF。以下是完整的踩坑记录和对应的系统性解决方案。

根本矛盾：HTML流式排版（无限滚动）→ A4分页PDF（固定页面），两种排版逻辑天然冲突。每次以为"调好了"，其实只是解决了当前版本的某个症状，没解决系统性问题。

踩过的坑（按严重程度排序）：

#坑症状根因解决方案
1 结构未拆干净 内容跑出边距、页与页之间空白、flow-content被截断 用div.page > div.page-inner做分页容器，改成流式排版时只patch了第一个位置（Q4），漏了Q7、Q10、开放问题的容器。patch一次不检查全部。 全量检查所有匹配项。改HTML结构不能只patch一处，必须搜索整个文件确认所有div.page都处理了。
2 @media print 隐藏规则 屏幕上看有18mm边距，打印/生成PDF时内容贴边、铺满整页 模板里.flow-content @media print{padding:0;width:100%}。屏幕显示正常，但打印时@media print优先级最高，把边距清零了。 改PDF排版第一件事：检查@media print规则。PDF专用CSS独立定义，不和屏幕样式混在一起。
3 封面字渲染糊了乱调CSS 封面白字在深蓝渐变上渲染有锯齿，以为是字不够粗/对比度不够 加了font-weight:800、text-shadow、letter-spacing——越改越糟。根因是Playwright emulate_media('print')对白字+深色渐变的渲染有锯齿，跟CSS样式无关。 封面糊 → 截图嵌入，不动CSS。遇到渲染问题先确认是否工具本身限制，不要盲目调样式。
4 PDF生成参数来回变 v6/v7正常，改参数后渲染异常 以为"越大越清晰"，改成viewport=2480×3508, scale_factor=3，结果反而不如1280×960, scale_factor=2。 生成参数定了就别动。锁定viewport=1280×960, scale_factor=2，不走回头路。
5 文件管理混乱 中间版本覆盖了源HTML，只能从模板重来 没有版本号管理，直接修改源文件。 每次修改前备份，或用v1/v2命名保留版本。最终版本归档为模板，下次只改内容不动结构。

系统性解决方案：分三层处理（内容/样式/输出独立）：

层做什么注意事项
① 纯内容HTML 只做内容分段，不关心样式和分页。div套div把数据结构化。 关注逻辑结构，不写任何排版相关的CSS。这是PDF的骨架。
② PDF专用CSS 独立定义样式。明确声明 @page { margin: 18mm 20mm; }，@media print里只放PDF需要的规则。 不和屏幕样式混在一起。@media print的规则优先级最高，必须主动检查。
③ 封面+封底独立处理 封面用固定CSS class，如果渲染糊→截图嵌入。封底用 page-break-before: always。 封面糊了不调CSS（根因是渲染引擎），直接截图嵌入。封底不能在flow-content里，必须独立分页。
📐 标准生成参数（锁定，不动）： viewport={'width': 1280, 'height': 960}, device_scale_factor=2 Playwright PDF参数：format='A4', print_background=True, margin={'top':'0','bottom':'0','left':'0','right':'0'} 🔒 生成前自查清单（强制检查，不通过不出PDF）： 1. 检查HTML里所有 div.page——预期只存在封面+封底2个 2. 检查 @media print 里没有 padding:0 / width:100% 等隐藏规则 3. 检查封面文字的CSS没有被改过（和模板diff对比） 4. 确认当前生成参数一致（viewport/scale_factor） 💾 归档原则：每个完成后，将最终版HTML + 生成脚本归档为模板。下次同类报告，从模板复制，只改内容，不动结构。

🤔 为什么需要知识库？

Agent能记住"怎么做"，但记不住"知道什么"

你可能会发现：Agent能记住你的沟通偏好（比如"说重庆话"），但当你问"通威去年做了多少培训"时，它需要重新翻文件、查数据。因为Agent记住的是关于怎么做的信息（偏好、工作流），而不是关于什么的知识（客户数据、行业动态、政策法规）。

知识库就是要解决这个问题——把"知道什么"也变成Agent可以随时调用的东西。

那知识库长什么样？怎么建？下面一步步来讲。

LLM Wiki是什么

Agent能读写的知识库，跟你共享同一个文件夹

LLM Wiki不是什么新软件——它就是一个文件夹，里面装着Markdown文件。跟等下要讲的Obsidian用的是同一个文件夹。区别只是：你通过Obsidian看，Agent通过LLM Wiki自动读写。

三个文件管好知识库

1️⃣ SCHEMA.md — 规矩文件：标签怎么打、文件名怎么起、每个分类放什么。Agent写文件时按规矩来，不会乱丢。
2️⃣ index.md — 目录索引：Agent每次读写前先扫一遍，知道有哪些知识可以用。
3️⃣ log.md — 操作日志：记录每次修改，谁、什么时候、改了啥，一目了然。

知识怎么分类——四个核心分类

分类回答的问题举例
entities/ 实体谁？什么？通威是谁？团队有哪些人？销售数据是什么？
concepts/ 概念是什么？TVET是啥？NP怎么算？AI出海是什么？
comparisons/ 对比哪个好？SGS vs TÜV对比、2025 vs 2026对比
queries/ 查询存档查出过什么？以前查过的有价值的问答结果
跟L盘的区别：L盘是按"文件类型和管理流程"存 → 适合做台账。LLM Wiki是按"知识本质和关联"存 → 适合查知识。两个不冲突，L盘管流程，Wiki管知识。
一句话：LLM Wiki就是"给Agent看的说明书"——告诉Agent知识放哪、按什么规矩写、怎么找到。你跟Agent共享同一个文件柜。

🖥️ Obsidian是什么

一个本地Markdown笔记软件，但远不止"记笔记"

刚才说LLM Wiki是一个Markdown文件夹，那你用什么来看它？就是Obsidian。Obsidian是一个本地优先的Markdown笔记工具，你写的每一个笔记都是.md文件存在你电脑上，不上传任何云端。但Obsidian最厉害的地方不是记笔记，而是用链接把知识串起来。

它跟其他工具有什么不一样？

工具文件格式数据在哪能串知识吗
记事本.txt本地🔴 不行
Word.docx本地🔴 独立文档
Notion自有格式☁️ 云端🟡 需要网络
Obsidian.md本地 ✓🟢 链接+图谱，离线可用

Obsidian特有的三个能力

1️⃣ 双向链接：用 [[页面名]] 就能链接到另一页，被链接的页面会自动显示"反向链接"
2️⃣ Graph View（图谱视图）：所有链接关系变成一张网络图，一眼看清知识之间的关联
3️⃣ 插件生态：社区有上千个插件，想要什么功能装一个就行

一句话：如果知识是一堆珠子，Obsidian就是那根线——帮你把珠子串起来。而且线在你手里（本地），不在别人服务器上。

🔗 知识库三件套的应用

三个工具各司其职，配合起来就是一个知识管理闭环

各自的角色

工具角色对应你日常中的什么
LLM Wiki 知识库本体全部.md文件，存数据、写笔记的地方
Obsidian 🖥️阅览室+编辑器你看文件、搜内容、建链接、写笔记的界面
Hermes 自动化管家自动更新知识、自动查找回答、帮你归档

三件套的配合闭环

① 信息进来（Hermes采集）
不管是RSS读到的新政策、你发我的行业文章，还是会议纪要里的关键数据——我扫到有用的，就按规矩写到Wiki里。

② 知识沉淀（LLM Wiki存储）
写好的内容按分类放好：客户数据在entities/、概念在concepts/。SCHEMA.md保证不乱，index.md保证找得到。

③ 你查看和补充（Obsidian浏览+编辑）
打开Obsidian就能看到所有新内容。你觉得不全的，直接改；你想补充观点的，直接写。不需要通过我。

④ 你喊一声，我再去读（Hermes查询）
不过这里有个实际操作的细节：我不会自动去翻Wiki目录。虽然SOUL.md里写了"优先读Wiki"，但我不会每次对话都去扫一遍所有.md文件。你编辑完Obsidian之后，需要跟我说一句\"八戒，你去读一下Wiki里的XX文件\"，我才会去读，然后就能用上最新内容了。

一句话总结这个闭环：Hermes往里放 → LLM Wiki存好 → Obsidian给你看和改 → 你告诉我再去读。知识只会越用越多，不会越用越散。
⚡ 实操关键：SOUL.md写规则 ≠ 自动执行

你在SOUL.md里写了"数据优先走Wiki"这个规则，但Agent不会因为这个规则就自动去扫描Wiki文件夹。启动读取的那一步，需要你主动触发——说一句"去看"就行了。这不是Bug，是设计：Wiki文件可能很多，全部自动加载会浪费算力，你指定具体文件才高效。

跟以前比有什么不一样？

以前：信息在邮件里→用一次→忘了→下次再翻
现在：信息进Wiki→随时查→随时补充→越积越厚

🚀 Hermes接管Obsidian（Local REST API）

从文件写入到API接管——三件套的最后一环打通了

为什么需要接管？

刚才讲了Hermes会往Wiki里写数据，但怎么写的？以前的方式是文件系统写入——Hermes直接往文件夹里塞.md文件。听起来简单，实际用起来问题不少：

问题表现
write_file建空壳文件创建了但内容是空的，或者内容不全
中文路径乱码文件夹或文件名含中文时就崩溃
不同步Hermes写进去了，但你在Obsidian看不到，需要手动刷新
不能删不能改名文件系统权限问题，只能写不能管理
Git Bash隔离Windows上的终端环境导致路径处理不一致

一句话：文件写入这条路，Windows上处处是坑。

解法：Local REST API 插件

最可靠的路径不是「更接近硬件」，而是「更接近应用层」。系统是一道墙，HTTP反而是一扇门。

Obsidian社区有个插件叫 Local REST API（作者：coddingtonbear），它在你的Obsidian上开一个本地HTTP服务（端口27124），其他程序（包括Hermes）可以直接通过API读写你的笔记。关键是——写入即同步，因为写操作直接走Obsidian的索引器，不是偷偷塞文件。

三级写入策略

考虑到API不是100%可用（Obsidian关了就没API了），设计了三级兜底：

优先级方式效果条件
🥇 第一优先Local REST API写入即同步，最快最稳Obsidian运行中
🥈 第二优先文件系统写入能写但不能同步，需手动刷新API不通时自动降级
🥉 第三优先通知你手动处理告诉你写失败了，你来处理两条路都走不通

Tavily是一个专为AI Agent设计的搜索引擎API。配置好API Key后，它能直接返回结构化的搜索结果，比手动打开浏览器搜索快10倍。典型用法：

浏览器搜索的适用场景

当Tavily搜不到足够信息时，用Bing搜索引擎配合浏览器工具手动搜索。使用browser_navigate(搜索URL)打开搜索结果页，不搜百度（被反爬严重），优先用Bing。

实战升级：2026年6月1日 — 引入AnySearch作为补充搜索

运行一段时间后，Tavily的免费额度在密集搜索时偶尔会耗尽（5月30日出现432超限）。同时深度研究场景下（如一次搜一家竞对的多个维度），Tavily需要多次调用。为此引入了 AnySearch 作为补充搜索工具。

① AnySearch是什么

AnySearch是一个面向AI Agent的搜索引擎API，与Tavily定位互补。它的三个差异化能力：

能力说明适合场景
POST /v1/search 自带content字段搜索时直接返回页面正文内容，不需要第二步提取需要原文深度阅读的研究
匿名免费层不配API Key也能用（每日IP限频），配Key后额度更高日常零散查询、节省Tavily额度
多参数筛选（zone/language/domains/freshness）可指定搜索区域（cn/intl）、语言、域名、时效定向搜索中国区特定信息

② AnySearch vs Tavily 使用分工

场景走哪个原因
每日新闻简报（cron job）Tavily已跑通、稳定、中文搜索结果精准
日常快速查询（\"帮我搜一下XX\"）AnySearch免费层不花钱、够用
竞对深度分析（一次性搜多个维度）AnySearch deep自动扩展query、多维度并行搜、融合结果
搜完马上要原文内容AnySearch直接返回正文，不用分两步搜+提取
Tavily超限/不可用自动fallback到AnySearch保证搜索不中断

③ 配置AnySearch

在 ~/.hermes/.env 中加入一行：

ANYSEARCH_API_KEY=as_sk_xxxx

然后创建一个Python搜索脚本，让Agent在需要时调用：

#!/usr/bin/env python3
"""AnySearch API 搜索工具"""
import os, json, urllib.request

ANYSEARCH_API_KEY = os.environ.get("ANYSEARCH_API_KEY", "")
API_URL = "https://api.anysearch.com/v1/search"

payload = {
  "query": "搜索关键词",
  "max_results": 5,
  "language": "zh-CN",
  "zone": "cn",
}
headers = {"Content-Type": "application/json"}
if ANYSEARCH_API_KEY:
  headers["Authorization"] = f"Bearer {ANYSEARCH_API_KEY}"
data = json.dumps(payload).encode("utf-8")
req = urllib.request.Request(API_URL, data=data, headers=headers, method="POST")
with urllib.request.urlopen(req, timeout=30) as resp:
  result = json.loads(resp.read().decode("utf-8"))

④ 关键原则

🐷 八戒实战教训：
① Tavily是默认后端，AnySearch是补充工具，不是替换。 Hermes的web.backend始终指向Tavily，AnySearch不走系统后端，通过独立脚本调用——两者独立运行，互不影响。
② 搜索后端跟模型无关。 无论Agent用的是DeepSeek还是Kimi，搜索配置都一样——模型的切换不影响搜索能力。
③ 匿名免费层有每日限额，但日常零散查询够用。 配了API Key后额度更高，更适合批量/深度搜索。

⚡ 关键实战教训

从真实调研中总结出来的要点

① 采集方法论有时效性

上午写"浏览器是唯一可靠手段"，下午Tavily API配好了就变成"首选Tavily"。采集环境是动态变化的，方法需要随时更新。

② 分析视角不能太窄

分析竞对时不要只盯着培训看。莱茵不是"培训公司"，是技术服务供应商。必须覆盖培训+咨询+认证三个维度，还要看对方集团全貌（如SGS的检测业务体量远大于培训）。

③ 不要用推断代替抓取

宁可标注"未找到公开数据"，也不根据行业通用情况推断岗位数量、薪资水平、团队规模。报告里只放真实抓取到的数据。

④ 双源交叉验证

同一个竞对的信息可能来自多个来源（课程商店 vs 集团官网 vs PDF课表）。获取到数据后必须对比差异，取最新/最官方的表述。三个来源的差距可能告诉你重要的信息——比如某个业务线在调整。

⑤ 报告不写采集方法

报告只呈现业务分析结果，不写采集过程、工具选择、难度评估。采集技巧和经验沉淀到Skill的参考资料中。June对"报告里夹带采集说明"很敏感。

一句话：市场调研的核心不是工具——是先定方向、再选路径、最后做报告。工具只是手段，回答你的业务问题才是目的。

第十二章 · AI Agent四引擎配置实战
Hermes + DeepSeek + Kimi + Codex 四引擎架构的实战搭建——如何规划模型分工、配置接入、协同工作流与实战教训

背景：2026年6月，June为八戒搭建了四引擎AI Agent体系。这源于一个判断：没有单一模型能同时做好分析、阅读、编码和总控。DeepSeek便宜够用但中文长文弱，Kimi长文强但编码弱，Codex能写代码但需要接入GPT额度。最终架构是Hermes做总控调度，各司其职。

① 四引擎架构总览

2026.06.02 · 八戒整理
引擎角色核心能力使用场景
Hermes总控调度器工具调用、任务编排、记忆系统、定时任务日常交互、信息采集、报告编排、知识库管理
DeepSeek主力分析推理能力强、API便宜、支持128K上下文数据分析、战略判断、竞对分析、日常问答
Kimi中文长文档中文理解最佳、PDF/财报长文提取、32K-128K上下文读政策报告、行业研报、合同条款、技术文档
Codex自动化工兵批量扫描、数据清洗、HTML生成、脚本执行批量文件处理、Excel解析、图表生成、格式检查
核心原则：不追求单一模型的全能，追求四引擎各司其职的最优组合。Hermes负责"知道交给谁最合适"。人负责做最终判断。

② DeepSeek：主力分析引擎

定位：日常主力 · base_url: api.deepseek.com/v1

DeepSeek作为八戒的默认模型，承担85%以上的日常交互。选择原因：推理能力接近GPT-4级别但API价格低一个数量级。在实际使用中表现稳定——能处理复杂的数据分析、战略判断和多轮推理。

优势劣势
推理能力强，适合结构化分析中文长文本理解不如Kimi
API便宜，可大量使用不心疼输出有token限制，超长报告需分节
支持128K上下文窗口特殊时期可能API不稳定

③ Kimi：中文长文档引擎

定位：备用和长文 · base_url: api.moonshot.cn/v1

Kimi作为第二引擎接入，专门处理DeepSeek不擅长的场景：中文政策报告、行业研报、财报的深度阅读和关键信息提取。选型时对比了Kimi K2.6和通义千问，Kimi的中文理解在实测中表现更好。

优势劣势
中文长文理解能力一流编码和结构化分析弱于DeepSeek
支持128K上下文，可读整份PDFAPI调用有一定频率限制
国内合规，不碰数据出境问题生态不如DeepSeek成熟
配置要点：Kimi国内版必须用kimi-coding-cn provider + moonshot.cn域名，而不是国际版。注册provider后通过hermes model交互菜单切换，不需要手动改config.yaml。

④ Codex：自动化工兵

定位：本地自动化工兵 · OpenAI OAuth接入

Codex（OpenAI CLI）的定位是八戒的"助理护士"——不负责分析判断，负责执行。它能批量扫描文件、读Excel/CSV、跑脚本、生成HTML图表、检查文件格式。Codex使用的GPT额度与ChatGPT Plus共用。

能做的事不能做的事
批量扫描文件和分类独立做业务判断和战略分析
读Excel/CSV并生成图表自动覆盖或删除Wiki知识库内容
执行本地脚本和自动修复决定Action Owner和任务优先级
检查文件格式和完整性在未授权的情况下修改配置文件
配置要点：Codex CLI通过npm install -g @openai/codex安装到系统PATH，然后用codex login --device-auth（手机扫码授权）完成ChatGPT账号绑定。默认model_provider为deepseek时需要手动改~/.codex/config.toml切换为openai。详细配置见第十一章"当前Skill状态"。

⑤ 协同工作流

四引擎不是各自独立使用的——它们通过Hermes的delegate_task机制协同工作。一个典型的信息分析任务流程如下：

步骤执行引擎做什么
1. 接收任务Hermes（八戒）理解需求、拆解任务、规划执行
2. 信息采集Hermes + search多来源交叉搜索、去重、初步分类
3. 长文阅读Kimi读取PDF/政策报告/长篇研报，提取关键信息
4. 数据分析DeepSeek逻辑推理、趋势判断、报告撰写
5. 自动化执行Codex批量处理、数据清洗、图表/HTML生成
6. 最终决策人（June）审阅输出、拍板定稿、决定下一步
关键经验：四引擎协同的瓶颈不在于AI能力，而在于"人要知道什么时候用哪个引擎"。June的经验是：大多数日常任务DeepSeek就够了；碰到长篇中文文档时主动切Kimi；需要批量处理时甩给Codex。引擎多了反而容易选择困难——先让一个跑通，再加第二个。

⑥ 实战教训

#教训说明
1不要同时配太多引擎June一开始想配齐DeepSeek+Kimi+Codex+OpenAI+Claude。实战发现两个主力（DeepSeek一个、Kimi一个备用）加Codex做自动化就够了。多一个引擎多一层配置成本和切换成本。
2provider注册 vs 模型切换config.yaml填好provider后，切模型用hermes model交互菜单，不要手动改yaml。Gateway也要读.env里的Key——项目级和用户级都要有。
3Codex的默认provider陷阱npm安装Codex后默认model_provider是deepseek。需要改~/.codex/config.toml里的model_provider = "openai"才能用GPT。保留一个deepseek.profile备份以防万一。
4API成本控制DeepSeek够便宜，Kimi也便宜，Codex走ChatGPT Plus额度。总成本可控。不建议接入多个商用模型（GPT API按量计费）——用Plus/Codex的OAuth额度就够了。
5不要"技术炫技"四引擎架构最终目的是帮June做业务——写方案、做分析、跟客户聊。不是为了展示"我有四个AI"。每个引擎的引入都必须回答一个问题：这个引擎能带来什么之前没有的能力？

① Survey Designer Skill — AI辅助问卷设计的基础设施

在开始设计前，先让Codex为Hermes创建了一个BS.A Survey & Interview Designer Skill（问卷与访谈设计专家），覆盖以下能力：

  • 问卷类型：客户需求调研、培训需求诊断、训前/训后问卷、客户访谈提纲、AI培训需求调研、大客户发现访谈、Workshop评估

④ AI辅助问卷设计方法论

适用于培训诊断、客户调研、Workshop评估等场景

完整的设计流程（可复用）：

Step 1：建立基础设施（Skill）

用Codex为Hermes创建专用Survey Designer Skill，包含问卷类型库、设计流程、质量闸门、输出格式模板。Skill提供"不遗忘的框架"，不替代领域判断。

Step 2：输入业务上下文

将培训大纲MD（v2.0）完整输入：培训定位、目标分层、议程、Workshop分组方式、六大场景、种子卡。让AI理解"为什么要设计这份问卷"比"怎么设计"更重要。

Step 3：人校准领域知识

AI提供的通用选项需由人修正为领域特有——业务线→四大业务群、岗位→三分类（工程师/销售/TS支持）、AI工具→国内+海外+Agent全谱系。这是AI无法替代的"业务感知"。

Step 4：多轮迭代优化

Q3工具清单经4轮迭代（加Agent/删企业协作/加海外/查漏补缺），Q1-Q2经业务线+岗位修正，口头禅"June"去除。每个修正都是"AI出草案→人发现问题→AI修正"的循环。

Step 5：输出可复用产物

最终输出三件套——正式问卷（可复制进问卷星）、题目设计表（每题的测量目标和分析用途）、分析计划（交叉维度+关键指标）。这些产物可直接复用为下次问卷的起点。

④ 文字绑定 — 最重要的技法

让文字显示在形状和箭头上

这是使用Excalidraw最核心、也最容易出错的技术。文字不能直接写在形状属性里——必须用容器绑定（Container Binding）：

形状端（boundElements） 文字端（containerId） ┌──────────────────────────┐ ┌──────────────────────┐ │ { "type": "rectangle", │ │ { "type": "text", │ │ "id": "box1", │ ⬌─────➤ │ "id": "t_box1", │ │ "boundElements": [ │ 互相引用 │ "containerId": │ │ { "id": "t_box1", │ │ "box1", │ │ "type": "text" } │ │ "text": "开始", │ │ ] │ │ "originalText": │ │ } │ │ "开始" │ └──────────────────────────┘ │ } │ └──────────────────────┘

三种绑定模式：

模式形状端文字端用途
带标签的形状boundElements含text idcontainerId=形状id流程节点、模块标签
带标签的箭头boundElements含text idcontainerId=箭头id标注连接含义（如"通过"、"拒绝"）
独立文本无需绑定自由定位标题、说明、注释
⚠️ 头号错误：使用 "label": {"text": "..."} —— 这个属性在Excalidraw中不存在，会被静默忽略，导致形状画出来了但上面没有文字。必须用容器绑定！这是新手最常见的坑。

文字要素清单（每项都必须有）：

  • fontFamily: 1 — Virgil手绘字体，不加则显示默认字体
  • fontSize — 标题≥20，正文≥16，标注≥14
  • originalText — 必须与text字段一致，Excalidraw用它做文字重排
  • autoResize: true — 自动调整文本框大小
  • textAlign: "center" + verticalAlign: "middle" — 绑定文字居中（有containerId时Excalidraw自动居中）

#坑正确做法
1头号陷阱 用 "label": {"text":"..."}该属性不存在！必须用 containerId + boundElements 双重绑定
2绑定了 shape→text 但忘绑 text→containerId双向引用缺一不可——形状的boundElements指向文字，文字的containerId指回形状

7背景色不透明遮挡下层元素分层背景区必须设置 opacity: 35，否则下层节点完全不可见
8用emoji做文字Excalidraw的Virgil字体不支持emoji渲染，会显示为方框
9浅灰文字配白底#b0b0b0 在白色背景上对比度严重不足——最低用 #757575
10独立文字定位靠 textAlign独立文字的 x 是左边缘。想居中需手动计算：cx - (text.length × fontSize × 0.5) ÷ 2

对P0和P1问题自动修复——颜色替换、字体修正、间距调整、移除非合规CSS——不改变内容只修设计合规性。

Step 6：交付前最终闸门

重新跑5大质量闸门（颜色/字体/布局/AI痕迹/双语）——全部通过才交付。Health Score ≥ 18/24方可发出去。
